Semana 01

Se requiere planificar actividades dentro de un presupuesto limitado.

El software es intangible hasta que se tiene un producto funcional.

Los requerimientos pueden cambiar, lo que exige una administración cuidadosa de recursos y coordinación entre:

  • Proveedores
  • Administradores
  • Desarrolladores

Desafíos Comunes

Cambios constantes en los requerimientos
Limitaciones de presupuesto
Plazos ajustados y presión por entrega
Comunicación entre equipos multidisciplinarios

Proceso de Gestión de Requerimientos

1
Recolección de Requerimientos

Identificación de necesidades y expectativas de los stakeholders a través de entrevistas, encuestas y talleres.

2
Análisis de Viabilidad

Evaluación técnica, operativa y económica de los requerimientos identificados.

3
Documentación

Elaboración de documentos formales que describen los requerimientos de manera clara y medible.

4
Validación

Verificación con los stakeholders de que los requerimientos capturados reflejan sus necesidades reales.

5
Gestión de Cambios

Proceso estructurado para evaluar, priorizar e implementar cambios en los requerimientos.

Rol de los Stakeholders

Hitos

Puntos de revisión clave en el progreso del proyecto que permiten evaluar el avance y tomar decisiones importantes.

Técnicas de negociación

Para equilibrar restricciones y expectativas entre las diferentes partes interesadas.

Método MoSCoW para priorizar requerimientos:

Must have
Requerimientos críticos que deben estar en la versión final.
Should have
Importantes pero no vitales, pueden posponerse si es necesario.
Could have
Deseables pero no esenciales, de bajo impacto si se omiten.
Won't have
Requerimientos de menor prioridad, no se incluirán en esta iteración.

Buenas Prácticas

Transparencia

Promover la transparencia en el equipo para generar confianza.

Autoorganización

Fomentar la autoorganización del equipo para mejorar la eficiencia.

Reconocimiento

Celebrar logros para reforzar la motivación del equipo.

Semana 03-04

Enfoque en las metodologías ágiles y su aplicación en el desarrollo de software.

Técnicas de estimación y planificación

  • Planning Poker
  • Estimación por puntos de historia
  • Velocidad del equipo
  • Burn-down charts

Ciclo de vida del software (SDLC)

Modelo en Cascada
  • Fases secuenciales
  • Documentación extensiva
  • Poco flexible a cambios
Modelo Ágil
  • Iterativo e incremental
  • Flexible a cambios
  • Entrega continua de valor

Principios del Manifiesto Ágil

Individuos e interacciones sobre procesos y herramientas
Software funcionando sobre documentación extensiva
Colaboración con el cliente sobre negociación contractual
Respuesta ante el cambio sobre seguir un plan

SCRUM: Marco de Trabajo Ágil

Roles en SCRUM

  • Product Owner: Representa al negocio
  • Scrum Master: Facilita el proceso
  • Equipo de Desarrollo: Desarrolladores, testers, etc.

Artefactos

  • Product Backlog
  • Sprint Backlog
  • Incremento

Eventos del Sprint

  1. Sprint Planning
  2. Daily Scrum
  3. Sprint Review
  4. Sprint Retrospective

Beneficios

  • Mayor visibilidad del progreso
  • Mayor flexibilidad
  • Mejor calidad del producto
  • Mayor satisfacción del cliente

Maxi Market - Supermercado en Línea

Desarrollo de una plataforma de supermercado en línea utilizando Extreme Programming (XP) como metodología ágil. Maxi Market es una solución innovadora que permite a los usuarios realizar compras de supermercado de manera rápida y sencilla desde cualquier dispositivo.

Características Principales

  • Catálogo de Productos: Amplia variedad de productos organizados por categorías
  • Carrito de Compras: Sistema intuitivo para gestionar los productos seleccionados
  • Chat en Vivo: Asistencia en tiempo real para los clientes
  • Gestión de Pedidos: Seguimiento en tiempo real del estado de los pedidos
  • Perfiles de Usuario: Personalización de la experiencia de compra

Innovaciones

Chat Box Inteligente

Sistema de chat avanzado que permite a los clientes realizar consultas en tiempo real, obtener recomendaciones de productos y recibir asistencia personalizada.

Búsqueda Avanzada

Potente motor de búsqueda con filtros inteligentes para encontrar productos de manera rápida y eficiente.

Metodología XP

Este proyecto se desarrolló siguiendo los principios de Extreme Programming, con:

  • Programación en parejas
  • Desarrollo guiado por pruebas (TDD)
  • Integración continua
  • Entregas incrementales
  • Retroalimentación constante del cliente

Extreme Programming (XP)

¿Qué es Extreme Programming (XP)?

Metodología ágil enfocada en:

  • Adaptabilidad al cambio
  • Entregas rápidas y funcionales
  • Colaboración estrecha con el cliente
  • Ideal para entornos dinámicos con requisitos cambiantes

Valores Fundamentales

Valor Descripción
Comunicación Diálogo constante entre equipo y cliente. Evita documentación innecesaria.
Simplicidad Soluciones mínimas que funcionen. Se evita la sobreingeniería.
Retroalimentación Pruebas continuas y entregas frecuentes para detectar errores temprano.
Coraje Tomar decisiones difíciles, eliminar código innecesario, admitir errores.
Respeto Ambiente colaborativo y valoración mutua entre miembros del equipo.

Prácticas Fundamentales

Planning Game

Planificación colaborativa con el cliente

  • Release Planning: 3-6 meses
  • Iteration Planning: 1-3 semanas
Pair Programming

Dos programadores trabajando juntos

  • Driver: Escribe el código
  • Navigator: Revisa y propone mejoras
TDD

Desarrollo guiado por pruebas

  • Red: Escribir prueba que falle
  • Green: Hacer que pase
  • Refactor: Mejorar el código
Integración Continua

Integración diaria del código

  • Pruebas automáticas
  • Builds frecuentes
  • Despliegues continuos
Ventajas
  • Alta calidad del software
  • Flexibilidad ante cambios
  • Reducción de riesgos
  • Mejor comunicación
  • Código más mantenible
Desafíos
  • Requiere disciplina constante
  • Dependencia del cliente
  • Difícil de escalar
  • No todos se adaptan al Pair Programming
